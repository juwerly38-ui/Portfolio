01
프로젝트 개요
프로토타입 v0.9 단일 HTML SPA 백엔드 연동 전

목적

사내 통역사 리소스를 체계적으로 관리하고, 신청자가 원하는 일정에 통역 지원을 요청할 수 있는 통합 스케줄 관리 시스템입니다. 현재는 프론트엔드 프로토타입으로 구현되어 있으며, 이 문서는 향후 백엔드 API 연동 및 실서비스 전환을 위한 기준서입니다.

핵심 가치

  • 신청자가 통역사 가용 여부를 실시간으로 확인하고 신청 가능
  • 일반 / 긴급 신청을 자동 분류하여 적절한 통역사 리소스 배분
  • 관리자가 수동 개입 없이도 자동 배정 제안을 받아 승인
  • 정례 회의를 등록하면 자동으로 일정 생성하여 반복 신청 불필요
  • 인트라넷 신청 폼과 API 자동 연동으로 중복 입력 제거

02
역할(Role) 설계
🙋 신청자 (Requester)
  • 일별·주별·월별 스케줄 열람
  • 가용 슬롯 클릭 → 신청 모달 (또는 인트라넷 폼으로 이동)
  • 본인 신청 건 조회 및 상태 확인
  • 정례 신청 제출
  • 통역사 이름·상세 정보 비공개
  • 긴급 신청 제한(D+3 버퍼 적용)
🛠️ 관리자 (Admin)
  • 전체 스케줄·통역사 현황 열람
  • 예약 신청 승인/반려 (수동 통역사 지정 가능)
  • 통역사 CRUD
  • 통역사 가능시간 등록
  • 수기 블록 설정 (요일 기반)
  • 회의 직접 추가/수정/삭제
  • CSV 내보내기
🎤 통역사 (Interpreter)
  • 본인 일정만 열람
  • 가능시간 등록/수정
  • 오늘 가능시간 미등록 시 경고 배지
  • 일별 뷰에서 본인 슬롯 상태 확인
  • 회의 추가/삭제 권한 없음

역할 전환 (프로토타입): 헤더 드롭다운에서 역할을 전환합니다. 실서비스에서는 사내 SSO/계정 시스템으로 role을 서버 세션에서 반환해야 합니다. 통역사 역할 선택 시 통역사 드롭다운이 추가 노출됩니다.

신청자 뷰에서의 표시 차이

신청자는 통역사 이름이나 회의명이 비공개입니다. 점유된 슬롯은 🔴 불가예상으로만 표시됩니다. 가용 슬롯은 🟢 협의 필요로 표시되어 클릭 시 신청 처리로 이어집니다.

06
신청 접수 플로우 — C안: 인트라넷 연동

C안 선택 이유: 신청서는 사내 인트라넷 폼에 이미 존재합니다. 스케줄 시스템에서 신청을 받는 대신, 해당 폼으로 사용자를 안내하고 인트라넷 폼 제출 시 API를 통해 스케줄 시스템에 자동으로 데이터가 전달되는 방식입니다. 중복 입력을 제거하고 기존 프로세스를 유지할 수 있습니다.

C안 전체 흐름도

sequenceDiagram autonumber actor User as 신청자 participant Sched as 스케줄 시스템 participant Intra as 인트라넷 폼 participant API as 스케줄 API 서버 participant DB as DB (reservations) User->>Sched: 가용 슬롯 클릭 Sched->>Sched: 슬롯 정보 파악\n(date, time, isEmergency) Sched->>User: 인트라넷 폼 URL로 리다이렉트\n?date=YYYY-MM-DD&time=HH:MM&emergency=0 User->>Intra: 인트라넷 폼 자동 열림\n(날짜·시간 자동 채워짐) User->>Intra: 나머지 필드 입력 후 제출 Intra->>API: POST /api/reservations\n{date, time, title, requester, ...} API->>API: 필수값 검증\n긴급 여부 자동 판정 API->>DB: 예약 저장 (status: pending) API->>Intra: 200 OK + reservationId Intra->>User: 제출 완료 안내 Note over Sched,API: 스케줄 시스템은 폴링 또는 WebSocket으로\n신규 pending 건 감지 → 관리자에게 배지 표시

긴급 신청 & D+3 버퍼 계산 로직

버퍼 개념

D+3 영업일: 오늘(D)로부터 영업일 기준 3일 후까지만 긴급 신청이 가능합니다. 영업일은 한국 공휴일과 주말을 제외합니다. 일본 공휴일은 영업일로 간주합니다.

D+3 계산 플로우

flowchart LR A([오늘 D]) --> B[영업일 카운터 = 0] B --> C[다음날로 이동] C --> D{한국 공휴일\n또는 주말?} D -->|Yes| C D -->|No| E[영업일 +1] E --> F{카운터 == 3?} F -->|No| C F -->|Yes| G([openDate = 해당 날짜]) G --> H{신청 대상 날짜\n≤ openDate?} H -->|Yes → 오픈| I([긴급 신청 가능\n🔓]) H -->|No → 잠김| J([긴급 신청 불가\n🔒]) style I fill:#dcfce7,stroke:#86efac style J fill:#fef3c7,stroke:#fcd34d

예시 케이스

시나리오오늘(D)D+1D+2D+3 = openDateD+4
평일 연속월화 ✅수 ✅목 ✅ 오픈금 🔒
D+1이 공휴일화수(공휴일) ❌목 ✅금 ✅토(스킵) → 월 ✅ 오픈
주말 걸침목금 ✅토(스킵)일(스킵) → 월 ✅화 🔒

// D+3 영업일(마지막 오픈 날짜) 계산
function calcBufferOpenDate() {
  var d = new Date();
  var count = 0;
  while (count < 3) {
    d = addDays(d, 1);
    if (d.getDay() !== 0 && d.getDay() !== 6 && !isKrHoliday(dateStr(d))) {
      count++;
    }
  }
  return dateStr(d); // YYYY-MM-DD
}

// 해당 날짜가 긴급 신청 잠금 상태인지 판정
function isBufferLocked(date) {
  return date > calcBufferOpenDate(); // true = 잠김
}
